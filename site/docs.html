Documentation

Ormit is an EF Core–style ORM for TypeScript — DbContext, typed queries, change tracking, and snapshot migrations, on any SQL database.

Getting started #

Choose your database — the install command and engine setup below update to match, and your choice is remembered across the page.

Install the core plus your dialect. Requires Node.js 18+ and TypeScript 5+ (strict recommended).

pnpm add @ormit/core @ormit/sqlite
pnpm add @ormit/core @ormit/postgres
pnpm add @ormit/core @ormit/mysql
pnpm add @ormit/core @ormit/mssql

Define plain entity classes and a context — this part is the same for every database:

import { DbContext, type DbContextOptions, type ModelBuilder } from '@ormit/core';

class User { id!: number; name!: string; age!: number; }

class AppDb extends DbContext {
  users = this.set(User);
  constructor(o: DbContextOptions) { super(o); }
  protected onModelCreating(m: ModelBuilder) {
    m.entity(User, e => e.toTable('users').hasKey('id'));
  }
}

Then pick the engine for your database and open a context:

import { SqliteEngine } from '@ormit/sqlite';

const engine = new SqliteEngine('app.db');   // or ':memory:' for tests
engine.exec(`
  CREATE TABLE users (
    id   INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    age  INTEGER NOT NULL
  );
`);
const db = new AppDb({ engine });
import { PostgresEngine } from '@ormit/postgres';

// Accepts a connection string or a node-postgres PoolConfig.
const engine = new PostgresEngine('postgres://user:pass@localhost:5432/app');
await engine.exec(`
  CREATE TABLE users (
    id   SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    age  INTEGER NOT NULL
  );
`);
const db = new AppDb({ engine });
import { MysqlEngine } from '@ormit/mysql';

const engine = new MysqlEngine('mysql://user:pass@localhost:3306/app');
await engine.exec(`
  CREATE TABLE users (
    id   INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    age  INT NOT NULL
  );
`);
const db = new AppDb({ engine });
import { MssqlEngine } from '@ormit/mssql';

const engine = new MssqlEngine({
  server: 'localhost', user: 'sa', password: process.env.MSSQL_PASSWORD!,
  database: 'app',
  options: { encrypt: false, trustServerCertificate: true },
});
await engine.exec(`
  CREATE TABLE users (
    id   INT IDENTITY(1,1) PRIMARY KEY,
    name NVARCHAR(255) NOT NULL,
    age  INT NOT NULL
  );
`);
const db = new AppDb({ engine });

From here everything — querying, saving, relationships — is identical regardless of the engine:

const faraj = db.users.add(Object.assign(new User(), { name: 'faraj', age: 28 }));
await db.saveChanges();          // INSERT — faraj.id is written back

const adults = await db.users.where(x => x.age.gte(18)).toList();
Only the engine import + construction and your DDL dialect differ between databases. The DbContext, model, queries, and saveChanges() are portable — Ormit adapts the SQL (RETURNING vs OUTPUT vs insertId, LIMIT/OFFSET vs TOP/OFFSET…FETCH) for you.

Defining a model #

Entities are plain classes (POCOs). Configure them fluently in onModelCreating. Precedence is convention < decorator < fluent — you only configure what conventions can't infer.

m.entity(Post, e => {
  e.toTable('posts', 'blog');                        // table + optional schema
  e.hasKey('id');                                    // or composite: hasKey('a', 'b')

  e.property(x => x.title).hasMaxLength(200).isRequired();
  e.property(x => x.slug).hasColumnName('slug_key'); // physical column name
  e.property(x => x.views).hasType('number').hasDefault(0);
  e.property(x => x.createdAt).hasDefaultSql('now()').valueGenerated('onAdd');
  e.property(x => x.rowVersion).isConcurrencyToken();
  e.property(x => x.note).isRequired(false).hasComment('optional');

  e.hasIndex('slug').isUnique().hasName('ix_post_slug');
  e.hasQueryFilter(x => x.deleted.eq(false));        // applied to every read
  e.hasData({ id: 1, title: 'Seed', slug: 'seed' }); // seed row for migrations
});

Property options (chain on property(x => x.field)): hasColumnName, hasType, hasMaxLength, isRequired, hasDefault, hasDefaultSql, valueGenerated('onAdd' | 'onAddOrUpdate'), isConcurrencyToken, hasComment.

Composite keys, owned types & inheritance

// Composite primary key.
m.entity(OrderLine, e => e.hasKey('orderId', 'lineNo'));

// Owned types — value objects flattened into the owner's table (address_city, …).
m.entity(User, e => {
  e.hasKey('id');
  e.ownsOne(Address, x => x.address, a => a.property(p => p.city).hasMaxLength(80));
  e.ownsMany(Tag, x => x.tags);   // owned collection in its own table
});

// Table-per-hierarchy (TPH) — one table, a discriminator column per subtype.
m.entity(Dog, e => e.toTable('animals').hasKey('id').hasDiscriminator('kind', 'dog'));
m.entity(Cat, e => e.toTable('animals').hasKey('id').hasDiscriminator('kind', 'cat'));

Relationships

// One-to-many (and its inverse to-one).
e.hasOne(User, x => x.author).withMany(u => u.posts)
  .hasForeignKey('authorId')
  .onDelete('cascade');            // 'cascade' | 'setNull' | 'restrict' | 'noAction'

// One-to-one.
e.hasOne(Profile, x => x.profile).withOne(p => p.user).hasForeignKey('userId');

// Optional reference (nullable FK → setNull default).
e.hasOne(Team, x => x.team).withMany().hasForeignKey('teamId').isRequired(false);

// Many-to-many (implicit join, or an explicit join entity with payload).
e.hasMany(Role, x => x.roles).withMany(r => r.users);
e.hasMany(Role, x => x.roles).withMany(r => r.users, UserRole); // usingEntity
The model finalizes once per context class into an immutable, JSON-serializable ModelSnapshot (sorted keys, byte-stable) that the migrations differ consumes. Invalid models throw eagerly with an actionable OMT12xx code — see Diagnostics.

Decorators

Prefer decorators? @ormit/decorators replays into the same builder — call applyDecorators(model, [...]) at the top of onModelCreating, then override fluently if needed:

import { entity, key, column, hasMany, hasOne, applyDecorators } from '@ormit/decorators';

@entity({ table: 'users' })
class User {
  @key() id!: number;
  @column({ maxLength: 80 }) name!: string;
  @column({ concurrencyToken: true }) version!: number;
  @hasMany(() => Post, { foreignKey: 'authorId' }) posts!: Post[];
}

@entity({ table: 'posts' })
class Post {
  @key() id!: number;
  @column({ maxLength: 200 }) title!: string;
  @hasOne(() => User, { foreignKey: 'authorId' }) author!: User;
}

class AppDb extends DbContext {
  protected onModelCreating(m: ModelBuilder) {
    applyDecorators(m, [User, Post]);
  }
}

Value converters

hasConversion('name') maps a property to a column through a named converter, applied at the boundary: toProvider on write and in where filters, fromProvider when materializing. The name is all that's stored in the snapshot, so migrations stay byte-stable; the functions are supplied at runtime via DbContextOptions.converters.

import { jsonConverter, booleanNumberConverter, isoDateConverter, defineConverter } from '@ormit/core';

class Account { id!: number; tags!: string[]; createdAt!: Date; premium!: boolean; role!: Role; }

// Model: name the converter each property uses.
m.entity(Account, e => {
  e.hasKey('id');
  e.property(x => x.tags).hasConversion('json');       // string[]  ⇆ JSON text
  e.property(x => x.createdAt).hasConversion('iso');    // Date      ⇆ ISO-8601 text
  e.property(x => x.premium).hasConversion('bool');     // boolean   ⇆ 0/1
  e.property(x => x.role).hasConversion('role');        // enum      ⇆ smallint (custom)
});

// Runtime: register the functions. Built-ins ship for the common cases; author
// your own with defineConverter (fully typed on both sides).
const db = new AppDb({
  engine,
  converters: {
    json: jsonConverter,
    iso: isoDateConverter,
    bool: booleanNumberConverter,
    role: defineConverter<Role, number>({
      toProvider: (r) => r.valueOf(),
      fromProvider: (n) => n as Role,
    }),
  },
});

// Round-trips as model values; filters accept model values too.
account.tags = ['a', 'b'];
await db.saveChanges();                                  // stores '["a","b"]'
await db.accounts.where(x => x.premium.eq(true)).toList(); // compares against 1
null passes through untouched. A property naming a converter that isn't registered throws at context construction. The column's stored type is the provider type — model it accordingly (e.g. hasType / the migration column type).

Querying #

Every Queryable is immutable — each call returns a new query. The lambda runs once at build time against a typed Proxy that records an IR.

const rows = await db.users
  .where(x => x.age.between(18, 65).and(x.email.isNotNull()))
  .orderByDescending(x => x.createdAt)
  .skip(20).take(10)
  .select(x => ({ id: x.id, name: x.name.toLower() }))
  .toList();

Shaping

Chain where (ANDed), orderBy/orderByDescending then thenBy/thenByDescending for secondary sort keys, skip/take, distinct(), and select(x => ({ … })) projections.

const roster = await db.people
  .orderBy(x => x.team)
  .thenByDescending(x => x.age)   // only available after orderBy (like EF's IOrderedQueryable)
  .thenBy(x => x.name)
  .toList();

Terminals

toList, first/firstOrNull, single/singleOrNull, count, any, sum/avg/min/max, toPage(page, size).

Raw SQL & tracking

// Parameterized — interpolations are always bound, never string-concatenated.
const found = await db.users.fromSql`SELECT * FROM users WHERE name = ${name}`.toList();

const readonly = await db.users.asNoTracking().toList();     // skip identity map
const all = await db.users.ignoreQueryFilters().toList();    // bypass global filters

Saving & change tracking #

Track entities with add/attach/remove, then persist atomically. Ormit diffs snapshots to find changes, topo-sorts by the FK graph (parents before children on insert, the reverse on delete), and writes generated keys back.

const user = await db.users.find(1);   // identity map first, then a keyed query
user!.name = 'Renamed';                // tracked — only changed columns update
db.users.remove(other);
const affected = await db.saveChanges();

// Ambient transaction spanning multiple saves:
await db.database.transaction(async () => {
  db.orders.add(order);
  await db.saveChanges();
});
Mark a property .isConcurrencyToken() (e.g. a version or rowversion). If a tracked update or delete matches zero rows, saveChanges() throws ConcurrencyError.

Entries & state

Inspect or drive tracking through db.entry(entity) — the state machine is Detached / Added / Unchanged / Modified / Deleted:

const entry = db.entry(user);
entry.state;                    // 'Unchanged' | 'Modified' | …
entry.modifiedProperties();     // ['name']  — scalar props that changed vs. the snapshot
db.users.attach(detachedUser);  // track an existing row as Unchanged
db.users.add(newUser);          // Added; db.users.remove(user) → Deleted

Context lifecycle

Contexts are short-lived and not concurrency-safe. Use a pooled factory (and dispose when done):

import { createContextFactory } from '@ormit/core';

const factory = createContextFactory(AppDb, { engine, poolSize: 8 });

await factory.scoped(async (db) => {          // create → run → auto-release
  db.users.add(user);
  await db.saveChanges();
});

// or manually:
const db = factory.create();
try { /* … */ } finally { await factory.release(db); }
// contexts also implement Symbol.asyncDispose: `await using db = factory.create();`

For web servers, @ormit/adapters wires this per-request into Express, Fastify, and NestJS — see Web adapters.

Relationships & loading #

Configure 1:1, 1:N, and N:M with hasOne/hasMany. Load related data eagerly, explicitly, or lazily.

// Eager — split queries by default (no cartesian explosion).
const blogs = await db.blogs
  .include(x => x.posts)
  .thenInclude((p: Post) => p.comments)
  .toList();

// Explicit — load on demand.
await db.entry(post).reference('author').load();
await db.entry(blog).collection('posts').load();

// Lazy — opt-in, always awaited (no hidden sync getters).
const author = await db.lazyReference<User>(post, 'author').load();
const posts = await db.lazyCollection<Post>(blog, 'posts').load();

// Filtered + multiple includes.
const users = await db.users
  .include(x => x.posts)
  .include(x => x.roles)     // many-to-many, loaded as a split query too
  .toList();

Turn on the N+1 detector by constructing the context in diagnostics mode:

const db = new AppDb({ engine, diagnostics: true, onWarning: (w) => logger.warn(w) });
// … looping and loading each post's author one-by-one emits OMT2001.
Cascade delete — a navigation's onDelete behavior is honored on saveChanges(): cascade bulk-deletes children, setNull clears their FK. Removing a principal removes its dependents automatically.

Migrations #

Migrations come from diffing your model snapshot against the last committed one — never from introspecting the live database (ADR-006). That keeps them deterministic and mergeable. The flow: build the model → diff → emit a migration file & commit the new snapshot → run the migrator.

Share one model definition between your context and the migration tooling:

import { ModelBuilder, ModelSnapshot } from '@ormit/core';

export function defineModel(m: ModelBuilder) {
  m.entity(User, e => e.toTable('users').hasKey('id'));
  m.entity(Post, e => {
    e.toTable('posts').hasKey('id');
    e.hasOne(User, x => x.author).withMany(u => u.posts).hasForeignKey('authorId');
  });
}

// The context uses it…
class AppDb extends DbContext {
  protected onModelCreating(m: ModelBuilder) { defineModel(m); }
}

// …and so does the tooling — build an immutable, byte-stable snapshot.
const mb = new ModelBuilder();
defineModel(mb);
const model = ModelSnapshot.build(mb);

Diff & apply

import { diffWithDown, EMPTY_SNAPSHOT, snapshotData, Migrator } from '@ormit/migrations';

// First migration: diff the model against an empty baseline.
const { up, down } = diffWithDown(EMPTY_SNAPSHOT, snapshotData(model));

const migrator = new Migrator(engine, [{ id: '0001_init', up, down }]);
await migrator.up();                    // apply pending — idempotent, safe to run twice
console.log(await migrator.applied());  // ['0001_init']  (tracked in __ormit_migrations)
await migrator.down(1);                 // revert the last migration

Evolving the schema

For later changes, diff the committed snapshot against your current model. Each change becomes an operation with an automatic inverse:

import { readFileSync } from 'node:fs';
import { deserializeSnapshot } from '@ormit/core';

const committed = deserializeSnapshot(readFileSync('model.snapshot.json', 'utf8'));
const { up, down } = diffWithDown(committed, snapshotData(model));
// up:   [{ kind: 'addColumn', table: 'users', column: { name: 'age', … } }]
// down: [{ kind: 'dropColumn', table: 'users', column: 'age' }]

Emit a migration file

import { emitMigration } from '@ormit/migrations';

const { id, filename, source } = emitMigration('add age', up, down);
// filename → 20260101120000_add_age.ts  (write `source` into your migrations dir)

The emitted module is plain, hand-mergeable TypeScript:

// 20260101120000_add_age.ts
import type { MigrationOperation } from '@ormit/core';

export const id = '20260101120000_add_age';
export const up: MigrationOperation[] = [
  { kind: 'addColumn', table: 'users',
    column: { name: 'age', type: 'number', nullable: true,
              maxLength: null, defaultValue: null, defaultValueSql: null } },
];
export const down: MigrationOperation[] = [
  { kind: 'dropColumn', table: 'users', column: 'age' },
];

Repair a merge conflict

If two branches both edit the committed .snapshot.json, re-derive the canonical snapshot from the model (the source of truth):

import { writeFileSync } from 'node:fs';
import { repairSnapshot } from '@ormit/migrations';

const { snapshot, changed } = repairSnapshot(model, readFileSync('model.snapshot.json', 'utf8'));
if (changed) writeFileSync('model.snapshot.json', snapshot);  // resolved deterministically
SQLite migrations run in a transaction; MySQL commits DDL implicitly, so its migrations aren't atomic. Every operation set carries an auto-generated inverse, so down() always exists.

CLI #

@ormit/cli is the programmable core behind the ormit command. Inject your engine, model, committed snapshot, and known migrations, then call the verbs — it's fully unit-testable without a real filesystem or process:

import { existsSync, readFileSync, writeFileSync } from 'node:fs';
import { createCli } from '@ormit/cli';

const cli = createCli({
  engine,
  model,                                   // ModelSnapshot from defineModel()
  committedSnapshot: existsSync('model.snapshot.json')
    ? readFileSync('model.snapshot.json', 'utf8')
    : undefined,
  migrations,                              // Migration[] you've emitted so far
});

// `ormit migrations add ` — diff, then emit a migration + the snapshot to commit.
const { migration, snapshot, destructive } = cli.add('init');
writeFileSync(`migrations/${migration.filename}`, migration.source);
writeFileSync('model.snapshot.json', snapshot);
if (destructive) console.warn('⚠ this migration drops a table or column');

await cli.update();                        // `ormit database update`  → ['0001_init']
await cli.revert(1);                       // `ormit database update --down 1`
const { applied, pending } = await cli.list();  // `ormit migrations list`
const sql = cli.script();                  // `ormit script` — forward DDL as text
const { changed } = cli.repair();          // `ormit migrations repair`

A thin binary wraps these into the terminal verbs:

ormit migrations add "init"     # diff → migration file + snapshot to commit
ormit migrations list           # applied vs. pending
ormit database update           # apply pending (idempotent)
ormit database update --down 1  # revert the last
ormit script                    # print the forward DDL
ormit migrations repair         # re-derive the snapshot after a merge conflict

Plugins #

Register plugins on the context. The first-party plugins are built solely on the public OrmPlugin surface — configureModel, normalizerPasses, and lifecycle interceptors — so your own plugins have the same power.

import { softDelete, timestamps, multitenancy } from '@ormit/plugins';

const db = new AppDb({
  engine,
  plugins: [
    softDelete(),                                    // remove() → UPDATE isDeleted + global filter
    timestamps(),                                    // stamp createdAt / updatedAt on save
    multitenancy({ tenant: () => als.getStore()! }), // scope every read + stamp inserts
  ],
});

Writing your own

A plugin is just an object on the OrmPlugin surface. configureModel shapes the model, normalizerPasses rewrite queries (IR → IR), and interceptors hook the save/command lifecycle:

import type { OrmPlugin } from '@ormit/core';

function auditColumns(): OrmPlugin {
  return {
    name: 'audit',
    interceptors: {
      // Stamp updatedBy on every insert/update before ops are built.
      savingChanges(ctx) {
        for (const entry of ctx.entries) {
          if (entry.state === 'Added' || entry.state === 'Modified') {
            (entry.entity as any).updatedBy = currentUser();
          }
        }
      },
      // Observe every executed write command (e.g. for logging / OTel).
      commandExecuted(cmd, result) {
        logger.debug({ sql: cmd.sql, affected: result.affected });
      },
    },
  };
}

Hooks available: configureModel(model), normalizerPasses: NormalizerPass[], and interceptors savingChanges / savedChanges / commandExecuting / commandExecuted. The first-party plugins use nothing else.

Web adapters #

A DbContext is short-lived and per-request. @ormit/adapters creates a pooled context at the start of each request and disposes it at the end — for Express, Fastify, and NestJS. Build the factory once at startup:

import { createOrmitFactory } from '@ormit/adapters';

const factory = createOrmitFactory(AppDb, { engine, poolSize: 8 });

Express

import { ormitExpress } from '@ormit/adapters';

app.use(ormitExpress(factory));              // attaches req.db, disposes on response end
app.get('/users', async (req, res) => {
  res.json(await (req as any).db.users.toList());
});

Fastify

import { ormitFastify } from '@ormit/adapters';

fastify.register(ormitFastify(factory));     // decorates request.db
fastify.get('/users', async (req) => (req as any).db.users.toList());

NestJS

import { Module } from '@nestjs/common';
import { ormitNestProviders } from '@ormit/adapters';

@Module({
  providers: ormitNestProviders(AppDb, { engine }),   // singleton factory + REQUEST-scoped AppDb
  exports: [AppDb],
})
export class DataModule {}

// then inject AppDb into any request-scoped provider:
constructor(private readonly db: AppDb) {}

Prefer to manage the lifecycle yourself? openScope(factory) returns { db, dispose }, or use the pooled factory directly — factory.create(), factory.release(db), factory.scoped(work).

Testing #

@ormit/testing ships an in-memory engine with real query semantics — filters, ordering, paging, aggregates, and writes all execute in-process, so unit tests need no database and no Docker.

import { InMemoryEngine } from '@ormit/testing';

const engine = new InMemoryEngine();
engine.seed('users', [
  { id: 1, name: 'Amal', age: 30 },
  { id: 2, name: 'Bilal', age: 17 },
]);

const db = new AppDb({ engine });
const adults = await db.users.where(x => x.age.gte(18)).toList();  // [Amal]

// Assert against executed commands and stored rows.
db.users.add(Object.assign(new User(), { name: 'Nour', age: 28 }));
await db.saveChanges();
expect(engine.rows('users')).toHaveLength(3);
expect(engine.log.length).toBeGreaterThan(0);   // every compiled command is logged
The in-memory engine is the same seam the dialects implement, so your queries and change tracking behave identically — it's ideal for fast unit tests, with Testcontainers reserved for dialect compatibility.

Recipe: Multi-tenant SaaS #

Scope every table by a tenant discriminator column, resolved per-operation from AsyncLocalStorage — the tenant id is never cached on the plugin instance, so one pooled context safely serves requests for different tenants.

import { AsyncLocalStorage } from 'node:async_hooks';
import { multitenancy } from '@ormit/plugins';

const tenantContext = new AsyncLocalStorage<string>();

const db = new AppDb({
  engine,
  plugins: [multitenancy({ tenant: () => tenantContext.getStore()! })],  // default column: tenantId
});

Resolve the tenant once per request, then run the rest of the handler inside it:

app.use((req, res, next) => {
  const tenantId = req.header('x-tenant-id')!;
  tenantContext.run(tenantId, next);
});

app.get('/projects', async (req, res) => {
  res.json(await db.projects.toList());   // WHERE "tenantId" = '<current tenant>' — automatic
});

app.post('/projects', async (req, res) => {
  db.projects.add(Object.assign(new Project(), req.body));
  await db.saveChanges();                  // tenantId stamped on the insert automatically
  res.sendStatus(201);
});
Scope only some entities with multitenancy({ tenant, entities: [Project, Invoice] }) — anything not listed is left untouched. Pair this with @ormit/adapters' ormitExpress() for the per-request context itself (see Web adapters).

Recipe: Soft delete & restore #

Model the flag column normally, then let the plugin rewrite deletes into updates and filter every read:

import { softDelete } from '@ormit/plugins';

m.entity(Post, e => {
  e.toTable('posts').hasKey('id');
  e.property(x => x.isDeleted).hasType('boolean').hasDefault(false);
});

const db = new AppDb({ engine, plugins: [softDelete()] });  // default column: isDeleted
db.posts.remove(post);
await db.saveChanges();                    // UPDATE posts SET "isDeleted" = true — no row is dropped

const visible = await db.posts.toList();                         // isDeleted = false, automatically
const everything = await db.posts.ignoreQueryFilters().toList();  // bypass the filter, see trashed rows too

The plugin only rewrites deletes — restoring is a normal tracked update through ignoreQueryFilters():

const trashed = await db.posts.ignoreQueryFilters().where(x => x.id.eq(postId)).single();
trashed.isDeleted = false;
await db.saveChanges();                    // UPDATE posts SET "isDeleted" = false
Scope with softDelete({ entities: [Post, Comment] }) — omit it and every entity declared in onModelCreating gets the filter. Pass column to use something other than isDeleted.

Recipe: Transactions & concurrency #

Wrap multiple saves in one atomic unit, and let a concurrency token protect against a stale write clobbering someone else's change.

m.entity(Account, e => {
  e.toTable('accounts').hasKey('id');
  e.property(x => x.version).isConcurrencyToken();  // bumped by your app, a trigger, or rowversion
});
await db.database.transaction(async () => {
  const from = await db.accounts.find(fromId);
  const to = await db.accounts.find(toId);
  from!.balance -= amount;
  to!.balance += amount;
  await db.saveChanges();   // both UPDATEs commit together, or neither does
});

A tracked update or delete that matches zero rows — because the row changed or vanished since it was read — throws ConcurrencyError instead of silently doing nothing:

import { ConcurrencyError } from '@ormit/core';

try {
  await db.saveChanges();
} catch (err) {
  if (err instanceof ConcurrencyError) {
    console.warn(`stale write on ${err.entries.length} entit${err.entries.length === 1 ? 'y' : 'ies'}`);
    // reload the affected entries and retry, or surface a conflict to the caller
  } else {
    throw err;
  }
}
Nested database.transaction() calls join the ambient transaction (via AsyncLocalStorage) instead of nesting — only the outermost call actually opens and commits a database transaction.

Recipe: Rich queries #

Compose filters, ordering, projection, and pagination — every call forks an immutable query, so you can build one up conditionally with plain ifs and no spooky mutation.

function search(db: AppDb, opts: { q?: string; minAge?: number; sort: 'name' | 'age' }) {
  let query = db.users.where(x => x.active.eq(true));

  if (opts.q) query = query.where(x => x.name.contains(opts.q!));
  if (opts.minAge) query = query.where(x => x.age.gte(opts.minAge!));

  const ordered = opts.sort === 'age'
    ? query.orderByDescending(x => x.age).thenBy(x => x.name)
    : query.orderBy(x => x.name);

  return ordered.select(x => ({ id: x.id, name: x.name, age: x.age }));
}

const page = await search(db, { q: 'am', sort: 'age' }).toPage(1, 20);

Scalar aggregates are query terminals, not part of the predicate — they run over the whole filtered set:

const paid = db.orders.where(x => x.status.eq('paid'));
const stats = {
  count: await paid.count(),
  revenue: await paid.sum(x => x.total),
  average: await db.orders.avg(x => x.total),
};

// Distinct values: .distinct() composes with toList(), not with a terminal
// aggregate — count the array rather than chaining .distinct().count().
const countries = await db.orders.select(x => ({ country: x.country })).distinct().toList();
console.log('distinct countries:', countries.length);
Filtering or counting through a to-many navigation from inside where() — x.posts.any(...), x.posts.count() — records into the IR but isn't lowered to SQL by the shipping generator yet (see the callout under Operators above). Reach for a query on the child entity instead, e.g. db.posts.where(x => x.published.eq(true)).select(x => ({ authorId: x.authorId })).distinct().

Recipe: Audit trail #

Write an audit row for every insert/update/delete, in the same transaction as the change that caused it — by adding to the tracker from inside savingChanges, before the write plan is built:

import type { OrmPlugin, SavingContext } from '@ormit/core';

class AuditLog {
  id!: number;
  entityName!: string;
  entityKey!: string;
  action!: 'Added' | 'Modified' | 'Deleted';
  changedBy!: string;
  changedAt!: Date;
}
// m.entity(AuditLog, e => e.toTable('audit_logs').hasKey('id')); — register it like any entity

function auditTrail(currentUser: () => string): OrmPlugin {
  return {
    name: 'audit-trail',
    interceptors: {
      savingChanges(ctx: SavingContext) {
        for (const entry of ctx.entries) {
          if (entry.entityName === 'AuditLog' || entry.state === 'Unchanged') continue;
          const key = String((entry.entity as { id?: unknown }).id ?? '');
          const log = Object.assign(new AuditLog(), {
            entityName: entry.entityName,
            entityKey: key,
            action: entry.state,
            changedBy: currentUser(),
            changedAt: new Date(),
          });
          ctx.tracker.track(log, 'AuditLog', 'Added');  // joins THIS saveChanges' write plan
        }
      },
    },
  };
}

const db = new AppDb({ engine, plugins: [auditTrail(() => currentUser().id)] });

db.users.add(user);
await db.saveChanges();   // writes `users` and one `audit_logs` row — atomically, one transaction
SavingContext.tracker is the same ChangeTracker the DbContext itself uses — anything you .track() onto it here is picked up by the same planSave() call, topo-sorted and executed alongside your own changes. No second saveChanges(), no separate transaction.

Recipe: Schema evolution #

The full loop — add a field, regenerate, re-apply, keep the data — is small enough to run in a minute. It's also a complete, runnable example in the repo: examples/migration-first.

# 1. Ship the current model as-is.
pnpm migrations:add init
pnpm db:update

# 2. Add a field to the model (e.g. `bio!: string | null;` on User,
#    configured with `e.property(x => x.bio).isRequired(false);`).

# 3. Diff against the *committed* snapshot — not the live database.
pnpm migrations:add "add user bio"
pnpm db:update            # applies only the new migration; existing rows keep their data

pnpm migrations:list      # applied vs. pending, by id
pnpm db:script            # print the forward DDL for every registered migration
Commit the emitted file under migrations/ together with the updated model.snapshot.json — they're one unit of change. If two branches both edit the snapshot and conflict in git, pnpm migrations:repair (wraps repairSnapshot()) re-derives it deterministically from the model rather than asking you to hand-merge JSON.
