Documentation

Ormit is an EF Core–style ORM for TypeScript — DbContext, typed queries, change tracking, and snapshot migrations, on any SQL database.

Getting started #

Choose your database — the install command and engine setup below update to match, and your choice is remembered across the page.

Install the core plus your dialect. Requires Node.js 18+ and TypeScript 5+ (strict recommended).

pnpm add @ormit/core @ormit/sqlite
pnpm add @ormit/core @ormit/postgres
pnpm add @ormit/core @ormit/mysql
pnpm add @ormit/core @ormit/mssql

Define plain entity classes and a context — this part is the same for every database:

import { DbContext, type DbContextOptions, type ModelBuilder } from '@ormit/core';

class User { id!: number; name!: string; age!: number; }

class AppDb extends DbContext {
  users = this.set(User);
  constructor(o: DbContextOptions) { super(o); }
  protected onModelCreating(m: ModelBuilder) {
    m.entity(User, e => e.toTable('users').hasKey('id'));
  }
}

Then pick the engine for your database and open a context:

import { SqliteEngine } from '@ormit/sqlite';

const engine = new SqliteEngine('app.db');   // or ':memory:' for tests
engine.exec(`
  CREATE TABLE users (
    id   INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    age  INTEGER NOT NULL
  );
`);
const db = new AppDb({ engine });
import { PostgresEngine } from '@ormit/postgres';

// Accepts a connection string or a node-postgres PoolConfig.
const engine = new PostgresEngine('postgres://user:pass@localhost:5432/app');
await engine.exec(`
  CREATE TABLE users (
    id   SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    age  INTEGER NOT NULL
  );
`);
const db = new AppDb({ engine });
import { MysqlEngine } from '@ormit/mysql';

const engine = new MysqlEngine('mysql://user:pass@localhost:3306/app');
await engine.exec(`
  CREATE TABLE users (
    id   INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    age  INT NOT NULL
  );
`);
const db = new AppDb({ engine });
import { MssqlEngine } from '@ormit/mssql';

const engine = new MssqlEngine({
  server: 'localhost', user: 'sa', password: process.env.MSSQL_PASSWORD!,
  database: 'app',
  options: { encrypt: false, trustServerCertificate: true },
});
await engine.exec(`
  CREATE TABLE users (
    id   INT IDENTITY(1,1) PRIMARY KEY,
    name NVARCHAR(255) NOT NULL,
    age  INT NOT NULL
  );
`);
const db = new AppDb({ engine });

From here everything — querying, saving, relationships — is identical regardless of the engine:

const nour = db.users.add(Object.assign(new User(), { name: 'Nour', age: 28 }));
await db.saveChanges();          // INSERT — nour.id is written back

const adults = await db.users.where(x => x.age.gte(18)).toList();
Only the engine import + construction and your DDL dialect differ between databases. The DbContext, model, queries, and saveChanges() are portable — Ormit adapts the SQL (RETURNING vs OUTPUT vs insertId, LIMIT/OFFSET vs TOP/OFFSET…FETCH) for you.

Defining a model #

Entities are plain classes (POCOs). Configure them fluently in onModelCreating. Precedence is convention < decorator < fluent — you only configure what conventions can't infer.

m.entity(Post, e => {
  e.toTable('posts', 'blog');                 // table + optional schema
  e.hasKey('id');                             // or composite: hasKey('a', 'b')
  e.property(x => x.title).hasMaxLength(200).isRequired();
  e.property(x => x.body).hasConversion('markdown');
  e.hasIndex('slug').isUnique();
  e.hasOne(User, x => x.author).withMany(u => u.posts).hasForeignKey('authorId');
  e.ownsOne(Address, x => x.address);         // flattened into prefixed columns
  e.hasQueryFilter(x => x.deleted.eq(false)); // global filter
  e.hasData({ id: 1, title: 'Seed' });        // seed row
});
The model finalizes once per context class into an immutable, JSON-serializable ModelSnapshot (sorted keys, byte-stable) that the migrations differ consumes. Invalid models throw eagerly with an actionable OMT12xx code.

Prefer decorators? @ormit/decorators replays into the same builder:

import { entity, key, column, hasMany, applyDecorators } from '@ormit/decorators';

@entity({ table: 'users' })
class User {
  @key() id!: number;
  @column({ maxLength: 80 }) name!: string;
  @hasMany(() => Post, { foreignKey: 'authorId' }) posts!: Post[];
}
// inside onModelCreating: applyDecorators(model, [User, Post]);

Querying #

Every Queryable is immutable — each call returns a new query. The lambda runs once at build time against a typed Proxy that records an IR.

const rows = await db.users
  .where(x => x.age.between(18, 65).and(x.email.isNotNull()))
  .orderByDescending(x => x.createdAt)
  .skip(20).take(10)
  .select(x => ({ id: x.id, name: x.name.toLower() }))
  .toList();

Raw SQL & tracking

// Parameterized — interpolations are always bound, never string-concatenated.
const found = await db.users.fromSql`SELECT * FROM users WHERE name = ${name}`.toList();

const readonly = await db.users.asNoTracking().toList();     // skip identity map
const all = await db.users.ignoreQueryFilters().toList();    // bypass global filters

Saving & change tracking #

Track entities with add/attach/remove, then persist atomically. Ormit diffs snapshots to find changes, topo-sorts by the FK graph (parents before children on insert, the reverse on delete), and writes generated keys back.

const user = await db.users.find(1);   // identity map first, then a keyed query
user!.name = 'Renamed';                // tracked — only changed columns update
db.users.remove(other);
const affected = await db.saveChanges();

// Ambient transaction spanning multiple saves:
await db.database.transaction(async () => {
  db.orders.add(order);
  await db.saveChanges();
});
Mark a property .isConcurrencyToken() (e.g. a version or rowversion). If a tracked update or delete matches zero rows, saveChanges() throws ConcurrencyError.

Contexts are short-lived and per-request. createContextFactory() pools and resets them; @ormit/adapters wires that into Express, Fastify, and NestJS.

Relationships & loading #

Configure 1:1, 1:N, and N:M with hasOne/hasMany. Load related data eagerly, explicitly, or lazily.

// Eager — split queries by default (no cartesian explosion).
const blogs = await db.blogs
  .include(x => x.posts)
  .thenInclude((p: Post) => p.comments)
  .toList();

// Explicit — load on demand.
await db.entry(post).reference('author').load();
await db.entry(blog).collection('posts').load();

// Lazy — opt-in, always awaited (no hidden sync getters).
const author = await db.lazyReference<User>(post, 'author').load();
In diagnostics mode Ormit's N+1 detector flags repeated single-entity loads of the same type (code OMT2001) and points you at Include.

Migrations #

Migrations come from diffing your model snapshot against the last committed one — never from introspecting the live database. That keeps them deterministic and mergeable.

import { diffWithDown, EMPTY_SNAPSHOT, snapshotData, Migrator } from '@ormit/migrations';

const { up, down } = diffWithDown(EMPTY_SNAPSHOT, snapshotData(model));
const migrator = new Migrator(engine, [{ id: '0001_init', up, down }]);

await migrator.up();      // apply pending — idempotent, safe to run twice
await migrator.down(1);   // revert the last migration

The @ormit/cli facade backs the intended ormit verbs:

ormit migrations add "init"     # diff → migration file + snapshot to commit
ormit database update           # apply pending (idempotent)
ormit database update --down 1  # revert
ormit migrations repair         # re-derive the snapshot after a merge conflict

Plugins #

Register plugins on the context. The first-party plugins are built solely on the public OrmPlugin surface — configureModel, normalizerPasses, and lifecycle interceptors — so your own plugins have the same power.

import { softDelete, timestamps, multitenancy } from '@ormit/plugins';

const db = new AppDb({
  engine,
  plugins: [
    softDelete(),                                    // remove() → UPDATE isDeleted + global filter
    timestamps(),                                    // stamp createdAt / updatedAt on save
    multitenancy({ tenant: () => als.getStore()! }), // scope every read + stamp inserts
  ],
});

Diagnostics #

Errors carry stable codes. Model validation runs eagerly and reports every problem at once.

CodeMeaning
OMT1001Expression could not be translated (thrown at build time)
OMT1003Concurrency conflict (ConcurrencyError)
OMT12xxModel validation (no key, duplicate table, bad FK, …)
OMT2001Possible N+1 (diagnostics mode)
